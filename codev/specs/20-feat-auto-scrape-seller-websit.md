## Clarifying Questions Asked

No clarifying questions needed — the issue (#20) provides a clear problem statement with acceptance criteria, and the existing codebase makes the integration points unambiguous. Key decisions derived from codebase analysis:

1. **Where does seller intelligence live?** — Extends the existing `SellerProfile` TypedDict (`backend/models/state.py`) and `SellerProfileConfig` Pydantic model (`backend/config/loader.py`), which currently holds `company_name`, `portfolio_summary`, and `portfolio_items`.
2. **How is scraping done?** — The codebase already has `backend/tools/web_crawler.py` (HTML-to-text extraction) used by the capability map generator. This same tool can be reused.
3. **Where does intelligence enter drafts?** — `backend/agents/draft.py` already injects seller profile via `_build_draft_system_prompt()`. Seller intelligence extends this injection point.
4. **Caching mechanism?** — Config is persisted to `~/.signalforge/config.json` via `backend/config/loader.py`. Seller intelligence should be cached alongside the existing seller profile in this same config store.

## Problem Statement

Outreach emails generated by SignalForge feel generic because the draft agent only has access to basic seller profile information: `company_name`, `portfolio_summary`, and `portfolio_items[]`. This is insufficient for crafting compelling, differentiated outreach.

Sales professionals need their outreach to reflect deep knowledge of their own product's unique value — key differentiators, proven sales plays, customer proof points, and competitive positioning. This information is typically available on the seller's public website but is not currently captured or used by SignalForge.

Without this intelligence:
- Drafts reference portfolio items generically instead of highlighting specific differentiators relevant to the prospect's pain point
- Sales plays cannot be matched to signal categories (e.g., a cost optimization signal should trigger a FinOps-related sales play)
- No customer proof points are available to build credibility
- Competitive positioning is absent, making drafts less persuasive in competitive deals

## Current State

**Seller Profile Today**:
- `SellerProfile` TypedDict: `company_name`, `portfolio_summary`, `portfolio_items[]`
- Configured during setup wizard (Step 1) or via Settings Panel (Seller Profile tab)
- Stored in `~/.signalforge/config.json` via `SellerProfileConfig`
- Injected into drafts via `_build_draft_system_prompt()` in `backend/agents/draft.py`

**Draft Generation Today**:
- System prompt includes: `"Seller context: {company_name} — {portfolio_items}."`
- Instruction: `"Only reference this if clear, specific connection. Do not force it."`
- No differentiators, sales plays, proof points, or competitive positioning available
- Solution areas from `solution_mapping` are vendor-agnostic by design — the bridge from vendor-agnostic areas to seller-specific value happens only through `portfolio_items`, which are just product names

**Existing Scraping Infrastructure**:
- `backend/tools/web_crawler.py` provides `crawl_url(url, timeout)` for HTML-to-text extraction
- Used by `capability_map_generator.py` for the `product_url` input mode
- Not used in the main pipeline or seller profile setup

## Desired State

**Enriched Seller Profile**:
- Seller profile includes a `website_url` field and structured `seller_intelligence` containing:
  - **Key differentiators**: What makes the seller's product/service unique
  - **Sales plays**: Common use cases and value propositions mapped to problem categories
  - **Proof points**: Customer logos, case study summaries, quantified outcomes
  - **Competitive positioning**: How the seller differentiates vs alternatives
- Intelligence is extracted from the seller's public website using LLM-powered analysis
- Intelligence is cached persistently (not re-scraped every session)
- Users can trigger re-scraping on demand and view/edit extracted intelligence

**Enriched Draft Generation**:
- Draft system prompt includes seller intelligence alongside existing portfolio items
- Differentiators are selected based on relevance to the persona's pain point and signal context
- Sales plays are matched to signal categories (e.g., cost optimization signal -> FinOps sales play)
- Proof points are included when relevant to build credibility
- Competitive positioning is available for drafts targeting prospects in competitive evaluations
- Graceful fallback to existing `portfolio_summary` behavior if scraping fails or intelligence is empty

## Constraints

### Technical Constraints
- Must reuse the existing `web_crawler.py` tool for HTML extraction (no new scraping dependencies)
- Must extend existing `SellerProfile`/`SellerProfileConfig` models (not replace them)
- LLM extraction must work with the configured `llm_provider`/`llm_model` from user settings
- Scraping happens during setup or on-demand — not during pipeline execution (to avoid latency and cost per session)
- Must handle websites that block crawlers, have CAPTCHAs, or return errors gracefully
- Website scraping should target key pages: homepage, product/solutions pages, case studies/customers, about page
- Page discovery should be best-effort from homepage links (not a full site crawl)

### Business Constraints
- Seller intelligence extraction is a one-time setup cost, not a per-session cost
- Must not break existing draft generation for users who don't provide a website URL
- Intelligence should be editable by users (LLM extraction is a starting point, not the final word)

## Assumptions
- Seller's public website contains meaningful information about differentiators, sales plays, and proof points
- The existing `web_crawler.py` tool can extract sufficient text from typical B2B SaaS/enterprise websites
- A single LLM call (or small number of calls) can extract structured intelligence from crawled page content
- Users are willing to provide their website URL during setup
- Cached intelligence remains valid for extended periods (weeks/months) — manual re-scrape is sufficient

## Solution Approaches

### Approach 1: LLM-Powered Website Intelligence Extraction (Recommended)

**Description**: Add a seller intelligence extraction agent that crawls key pages from the seller's website, combines the text content, and uses the configured LLM to extract structured intelligence into four categories. The extracted intelligence is stored as part of the seller profile config and injected into draft generation prompts.

**Pros**:
- Leverages existing `web_crawler.py` infrastructure
- LLM extraction produces high-quality structured output from unstructured web content
- Consistent with existing architecture (agents + LLM for intelligence extraction)
- One-time cost at setup, cached for all subsequent sessions
- Users can review and edit extracted intelligence

**Cons**:
- LLM extraction quality depends on website content quality
- Some websites may not be crawlable (SPAs, heavy JS rendering)
- Requires multiple page fetches (homepage + discovered subpages)

**Estimated Complexity**: Medium
**Risk Level**: Low — extends existing patterns, graceful fallback if scraping fails

### Approach 2: Manual Intelligence Entry Only

**Description**: Add structured intelligence fields to the seller profile but require manual entry instead of automated extraction.

**Pros**:
- No scraping infrastructure needed
- Intelligence is always accurate (human-entered)
- Simpler implementation

**Cons**:
- High friction for users — they must manually structure their own differentiators
- Defeats the purpose of automation
- Most users won't fill in detailed intelligence manually

**Estimated Complexity**: Low
**Risk Level**: Low — but low adoption risk

### Approach 3: Hybrid (Auto-Extract + Manual Edit)

**Description**: Same as Approach 1, but with stronger emphasis on the editing UX — presenting extracted intelligence as editable drafts that users refine.

**Pros**:
- Best of both worlds — automation for speed, manual editing for accuracy
- This is effectively Approach 1 with good UX defaults

**Cons**:
- Slightly more frontend work for inline editing

**Estimated Complexity**: Medium
**Risk Level**: Low

**Selected Approach**: Approach 1 (which naturally supports manual editing via the Settings Panel, making it equivalent to Approach 3 in practice).

## Open Questions

### Critical (Blocks Progress)
- [x] None — the issue description and codebase analysis provide sufficient clarity

### Important (Affects Design)
- [ ] How many pages should we crawl per website? Recommendation: Up to 10 pages (homepage + up to 9 discovered subpages matching patterns like /product, /solutions, /customers, /case-studies, /about)
- [ ] Should we support JavaScript-rendered SPAs? Recommendation: No — `web_crawler.py` uses basic HTTP fetch + HTML parsing. If a site requires JS rendering, the fallback is manual entry. This can be enhanced later.
- [ ] Should seller intelligence be versioned? Recommendation: No — a single cached version with a `last_scraped` timestamp is sufficient. Users can re-scrape to update.

### Nice-to-Know (Optimization)
- [ ] Could we periodically auto-refresh intelligence? Recommendation: Out of scope — manual re-scrape is sufficient for v1

## Performance Requirements
- **Scraping + Extraction Time**: < 60 seconds for a typical website (homepage + up to 9 subpages)
- **Draft Generation Impact**: Negligible — intelligence is pre-extracted and cached, not fetched at draft time
- **Storage**: Seller intelligence adds < 10KB to config file

## Security Considerations
- **URL Validation**: Website URL must be validated (HTTPS preferred, valid domain)
- **SSRF Prevention**: Only scrape user-provided URLs, not URLs discovered recursively beyond one hop from homepage
- **Content Sanitization**: Crawled content is passed to LLM as text (HTML stripped by web_crawler.py), no raw HTML execution
- **Data Privacy**: Seller intelligence is stored locally in `~/.signalforge/config.json`, same as existing config
- **Rate Limiting**: Respect robots.txt and add reasonable delays between page fetches

## Test Scenarios

### Functional Tests
1. **Happy path**: User provides website URL during setup → system scrapes and extracts structured intelligence → intelligence appears in Settings Panel → drafts reference relevant differentiators
2. **No URL provided**: User skips website URL → seller intelligence is empty → drafts fall back to existing portfolio_summary behavior → no errors
3. **Scraping failure**: Website is unreachable or blocks crawlers → graceful error message → intelligence remains empty → drafts use fallback
4. **Re-scrape**: User triggers re-scrape from Settings → new intelligence overwrites old → drafts use updated intelligence
5. **Manual edit**: User edits extracted intelligence in Settings → changes are persisted → drafts use edited values
6. **Draft enrichment**: Draft for cost-optimization signal references FinOps sales play → Draft for security signal references compliance differentiator
7. **Partial extraction**: Some intelligence categories are empty (e.g., no case studies on site) → non-empty categories still used → no errors

### Non-Functional Tests
1. **Timeout handling**: Slow website responds after crawler timeout → graceful degradation
2. **Large page content**: Very long pages are truncated before LLM extraction → extraction still succeeds
3. **Existing test suite**: All pre-existing tests continue to pass with no regressions

## Dependencies
- **External Services**: Seller's public website (HTTP/HTTPS)
- **Internal Systems**: `backend/tools/web_crawler.py` (HTML extraction), configured LLM provider

| Risk | Probability | Impact | Mitigation Strategy |

| Website not crawlable (SPA, CAPTCHA, blocks) | Medium | Low | Graceful fallback to manual entry; clear error message to user |

| Seller intelligence becomes stale over time | Low | Low | Manual re-scrape button; `last_scraped` timestamp shown in UI |

## Notes

The seller intelligence extraction is architecturally similar to the existing capability map generator (`backend/capability_map_generator.py`), which also scrapes a URL and uses LLM to extract structured data. The new seller intelligence agent can follow the same pattern but produce a different output structure tailored to sales enablement rather than signal matching.

Key architectural decision: Seller intelligence is stored as part of the seller profile config (alongside `company_name`, `portfolio_summary`, `portfolio_items`) rather than as a separate data store. This keeps the data model simple and ensures intelligence is always available wherever `seller_profile` is accessed.
