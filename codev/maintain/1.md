## Audit Goal

Scan all Python source under `backend/` and `tests/` to assess docstring
coverage, identify gaps that hurt onboarding/IDE help/OpenAPI output, and flag
any stale or misleading docstrings. No code changes in this phase — findings
feed the CLEAN phase.

## Method

1. Walked the AST of every `.py` file under `backend/` and `tests/`.
2. Counted module / class / function docstrings via `ast.get_docstring`.
3. Hand-spot-checked module docstrings to validate accuracy against the spec.
4. Searched for staleness markers (`TODO`, `FIXME`, `deprecated`) and verified
   that all referenced spec sections (`§5.x`, `§6.2`, etc.) still exist.

Tools used: `ast` (stdlib). `ruff check --select D` was not run because no
pydocstyle rules are configured in `pyproject.toml` and bulk auto-fix is out
of scope for AUDIT.

### Finding B — Public methods on `ConnectionManager` lack docstrings

**File**: `backend/api/websocket.py`

`ConnectionManager` defines 9 methods. Only `broadcast` has a docstring;
`__init__` does not need one. That leaves 7 public methods missing
docstrings: `connect`, `disconnect`, `broadcast_stage_update`,
`broadcast_pipeline_complete`, `broadcast_hitl_required`,
`broadcast_budget_warning`, `broadcast_error`. The module-level
`drain_event_queue` already has a docstring; the `_default` JSON-encoder
helper is nested inside `broadcast_stage_update` and is too small to need
one. The module docstring enumerates the event schemas, but per-method
docstrings would name the trigger and audience for each broadcast.

### Finding C — Public registry helpers lack docstrings

**File**: `backend/api/session_store.py`

`get_active_session`, `register_session`, `deregister_session`,
`generate_session_id` (lines 259–272). Single-line docstrings would suffice.
Their internal `_init_meta_db`, `_meta_db_path`, `_get_meta_session` helpers
also have no docstrings; the helper purpose is not obvious from the name
(`_get_meta_session` returns a SQLAlchemy session, not a SignalForge session).

### Finding D — Pipeline stage-node wrappers lack docstrings

**File**: `backend/pipeline.py` (lines 139–170)

`signal_ingestion_node`, `signal_qualification_node`, `research_node`,
`solution_mapping_node` — only `persona_generation_node` (L173) is
documented. These are LangGraph node functions; the module docstring shows
their place in the graph but each node would benefit from a one-liner stating
what it returns and which agent it dispatches to.

### Finding E — FastAPI route handlers in `settings.py` mostly undocumented

**File**: `backend/api/routes/settings.py` (8 of 19 functions documented)

Handlers without docstrings: `get_seller_profile`, `update_seller_profile`,
`get_seller_context`, `update_seller_context`, `get_api_keys`,
`update_api_keys`, `get_session_budget`, `update_session_budget`,
`get_langsmith`, `update_langsmith`, `_product_list_as_str`. Route
docstrings drive Swagger UI summaries — a one-line description per
endpoint is the standard FastAPI convention and is missing for most
settings endpoints.

### Finding F — Private LLM-prompt-building helpers lack docstrings

A consistent pattern across the agents: the private `_make_llm`,
`_build_*_prompt`, and `_*_to_text` helpers have no docstrings.

| File | Functions |
|---|---|
| `backend/agents/research.py` | `_make_llm` |
| `backend/agents/persona_generation.py` | `_make_llm`, `_make_persona` |
| `backend/agents/solution_mapping.py` | `_make_llm`, `_build_solution_mapping_prompt`, `_capability_map_to_text` |
| `backend/agents/synthesis.py` | `_build_synthesis_prompt` |
| `backend/agents/draft.py` | `_build_draft_user_prompt` |
| `backend/agents/signal_qualification.py` | `_normalized_llm_provider`, `_message_content_to_text` |
| `backend/capability_map_generator.py` | `_build_generation_prompt` |

For most, the function name is self-documenting (e.g., `_make_llm` returns
the configured LangChain LLM). Document only the non-obvious cases:
`_normalized_llm_provider` (what does "normalized" mean — strips suffix?
canonicalises to enum?), `_message_content_to_text` (which message types are
flattened, what happens to non-text content), and `_capability_map_to_text`
(format guarantees consumed by the prompt).

### Finding G — `document_parser.py` per-format helpers undocumented

**File**: `backend/tools/document_parser.py` (6 of 8 functions missing)

`_extract_pdf`, `_extract_docx`, `_extract_pptx`, `_extract_xlsx`,

contract; each helper is short and the format is in its name. **Leave
as-is** unless the CLEAN phase finds a non-obvious behaviour to document
(e.g., XLSX cell ordering, HTML script-tag stripping).
